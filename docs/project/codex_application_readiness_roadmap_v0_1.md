# Codex for Open Source Application Readiness Roadmap v0.1

## Status and purpose

- Status: Active evidence-accumulation roadmap
- Baseline release: `v0.1.0-rc.2`
- Application submission: Not authorized and not part of M6

This roadmap defines how the project will accumulate credible public OSS,
analysis, maintenance, and community evidence before a later Human decision on
whether to submit an application. Publishing one prerelease is a baseline, not
proof of sustained adoption or operational maturity.

## Principles

- Keep implementation, release, application readiness, and Human submission as
  distinct lifecycle states.
- Prefer verifiable repository, CI, Release, Issue, and case-study evidence over
  narrative claims.
- Record both successful operation and bounded defects or limitations.
- Publish only synthetic or explicitly public-safe evidence.
- Do not publish real exports, personal outputs, identifiers, paths, counts, or
  fingerprints.
- Do not create artificial Issues, feedback, users, or adoption claims.

## Stage 1 — Release baseline

Status: **Complete**

- Publish a reviewed GitHub prerelease from an exact CI-passing commit.
- Verify tag, Release metadata, public links, assets, tests, validators, and
  privacy after publication.
- Establish an evidence tracker and public release-operation record.

Completion evidence: `v0.1.0-rc.2`, M5.5B Phase 3 reports, and the M6
post-validation report.

## Stage 2 — Sustained OSS operation

Status: **Open**

- Accumulate factual release and maintenance history over multiple work cycles.
- Triage genuine Issues and record response, decision, fix, verification, and
  release linkage when they occur.
- Track CI reliability and any regressions without hiding failed runs.
- Preserve small, reviewable commits and current public documentation.
- Record community feedback only when it exists and can be cited publicly.

Readiness signal: evidence shows repeatable maintenance beyond the initial
release event.

## Stage 3 — Analysis and value evidence

Status: **Open**

- Add further synthetic or approved anonymous analysis cases that demonstrate
  distinct user questions and reproducible calculations.
- Keep Facts, Calculation, Interpretation, Uncertainty, and Unsupported
  Conclusions separate.
- Record which cases informed a product or documentation improvement.
- Preserve the local-first and external-sharing privacy boundary.

Readiness signal: multiple useful, reproducible cases show value beyond a
single demonstration without claiming private-user outcomes.

## Stage 4 — Application evidence review

Status: **Not started**

- Review the evidence tracker for completeness, public accessibility,
  provenance, and privacy.
- Confirm current license, rights, dependency, notice, security, release, and
  repository-health status.
- Identify unsupported claims and remove them from draft application material.
- Obtain an independent readiness review and resolve any rework.

Readiness signal: the evidence set is current, traceable, privacy-safe, and
sufficient to support every application claim.

## Stage 5 — Human application decision

Status: **Human-owned**

Human decides whether and when to finalize and submit an application. No
application form, message, or external submission is created or sent by this
roadmap.

## Review cadence

- Update the evidence tracker after each genuine release, Issue resolution,
  case study, community-feedback event, or material operational improvement.
- Reassess roadmap status at a bounded milestone rather than declaring
  readiness from elapsed time alone.
- Raise privacy, rights, public-contract, or evidence-integrity conflicts
  immediately; do not trade them for application speed.
